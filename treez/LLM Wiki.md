A knowledge base about **Treez Ecommerce** — the e-commerce platform built at Treez before it became GapCommerce. Maintained by an LLM from source code, documents, and conversations.

## The core idea

This wiki sits between you and the raw sources (repos, docs, transcripts, tickets). When a new source is added, the LLM doesn't just index it — it reads it, extracts what matters, and integrates it into the existing wiki: updating entity pages, revising summaries, flagging where new information contradicts old, strengthening the overall picture. Knowledge is compiled once and kept current, not re-derived on every query.

**The wiki is a persistent, compounding artifact.** The cross-references are already there. The contradictions have already been flagged. The synthesis already reflects everything ingested so far.

The LLM writes and maintains all of it. You're in charge of sourcing, direction, and asking the right questions. In practice: Claude Code open on one side, Obsidian open on the other. Claude makes edits, you browse the results in real time — following links, checking the graph view, reading the updated pages. Obsidian is the IDE; Claude is the programmer; this wiki is the codebase.

## Architecture

Three layers:

**Raw sources** — source documents the LLM reads but never modifies. Repo analysis documents, architecture writeups, meeting notes, decision logs, Slack threads, ticket exports. Source of truth.

**The wiki** — LLM-generated markdown files. Service pages, entity pages, concept pages, cross-repo comparisons, dependency maps, an overview, a running synthesis. The LLM owns this layer entirely.

**The schema** — `CLAUDE.md`, which defines wiki structure, conventions, and workflows. Co-evolved over time as we figure out what works for this domain.

## Operations

**Ingest.** Drop a new source into `raw/` and say "ingest". The LLM reads it, integrates it into the wiki — updating service pages, entity pages, the dependency map — and appends to the log. A single source might touch 10–15 wiki pages.

**Query.** Ask questions about Treez Ecomm — architecture decisions, data flow, which service owns what, what third parties are in use, what the pain points are. The LLM reads the index, drills into relevant pages, and synthesizes an answer. Good answers get filed back as wiki pages so they don't disappear into chat history.

**Lint.** Periodically ask the LLM to health-check the wiki: contradictions between service pages, stale claims superseded by newer sources, orphan pages, missing cross-references, gaps worth investigating.

## Indexing and logging

**index.md** — catalog of every wiki page, organized by category (services, entities, concepts, decisions, third-party dependencies, etc.), with a one-line summary per page. Updated on every ingest. The LLM reads this first when answering queries.

**log.md** — append-only record of ingests, queries, and lint passes. Each entry prefixed `## [YYYY-MM-DD] <type> | <title>` so it's greppable:
```
grep "^## \[" log.md | tail -5
```

## Source types

Sources relevant to this wiki:
- **Repo documents** — generated by running Claude Code inside each repo (see `CLAUDE.md` for the workflow)
- **Architecture docs** — any existing design docs, RFCs, or ADRs
- **Decision logs** — why things were built the way they were
- **Post-mortems / incident reports** — what broke and why
- **Meeting notes / Slack threads** — context that never made it into docs

## Why this works

The tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping. Updating cross-references, keeping summaries current, noting when new information contradicts old claims. LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass.

The human's job is to curate sources, direct the analysis, and ask good questions. The LLM's job is everything else.
